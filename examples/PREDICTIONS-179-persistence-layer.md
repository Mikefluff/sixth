# PREDICTIONS-179 — Persistence Layer for Discovered Candidates (cycle 35A)

**Date pre-registered:** 2026-05-24

**Attested via** `scripts/attest_prediction.sh` per Rule 9.
Initial attestation: see ledger row dated 2026-05-24.

---

## Cycle context

Cycles 25–34 built the **protocol** for substrate discovery
(DETECT-MOTIF-AUTO → SHADOW-CHECK → coupling → energy → HELD-OUT-EVAL
→ PROMOTE-STABLE) and the **metabolism** for surviving primitives
(decay → demote → decompose, dependency tracking, support credit).

Cycle 34C-bis verified the protocol works on fixtures and demo runs
but **L2 (substrate-discovered candidates) is empty** (count = 0):
no cand_NNN persists across runs, no `stdlib/promoted/` exists, no
`'promote-stable` events in the attestation ledger.

Cycle 35 is **the missing body** for discovered entities.  Without
it, every PROMOTE-STABLE is theatre that ends at process exit.

> **сначала орган, потом метаболизм.**

This pre-reg defines what cycle 35 must do AND what it must NOT
do.  The main pitfall is treating "make the directory" as
sufficient — it is not.

---

## CORE QUESTION (binding)

> Can the system mechanically produce, persist, and survive a
> substrate-discovered cand_NNN across at least one cross-boot
> metabolism cycle, with full lineage that any reviewer can audit?

---

## CORE INVARIANTS (binding, three rules)

1. **No persistent lineage, no L2.**  A cand_NNN with no
   `.lineage.json` and no ledger promote event does NOT count
   toward L2 occupancy, regardless of any in-process state.
2. **No production tag, no discovery.**  Only
   `source-tag = 'production` cand_NNN counts toward L2.
   Fixture and demo cands are explicitly excluded.
3. **No cross-boot survival, no durability.**  A cand_NNN that
   was PROMOTE-STABLE'd in one engine session but does not
   survive an engine boot (e.g., metabolism counter reset wipes
   it) is NOT durable.  Durability requires AT LEAST one
   successful boot-then-metabolism cycle.

These three invariants together are what distinguishes L2 from
a fixture pile with a fancy directory name.

---

## What cycle 35 is NOT

- Not a tuning of any cycle 25-34 mechanism.
- Not an introduction of new ontological primitives.  All cycle
  35 surface additions are L1 grammar verbs (persistence-write,
  load) or L4 implementation details (file paths, JSON schemas).
- Not a new metabolism rule.  Cycle 35 makes metabolism survive
  reboots; it does not change WHAT metabolism does.
- Not an unblock of cycle 34A.  34A remains blocked until L2
  is non-empty.
- Not a discovery-quality claim.  Cycle 35 only enables L2 to
  have occupants; whether the discoveries are "good" is a
  separate evaluation.

## 10 binding elements (every element MUST be specified before 35B impl)

### Element 1 — Exact persistence layout

```
stdlib/promoted/
├── <discovery_cycle>/
│   ├── cand_<NNN>.6th              ← Sixth source, immutable after write
│   ├── cand_<NNN>.lineage.json     ← provenance (see Element 4)
│   └── cand_<NNN>.metabolism.json  ← cross-boot momentum state (see Element 6)
├── _rolled-back/
│   └── cand_<NNN>.<timestamp>.json ← preserved lineage of decomposed cands
└── _index.json                      ← optional: hash-indexed lookup for boot load
```

Layout is binding.  Any deviation requires deprecation cycle.

### Element 2 — `PROMOTE-STABLE` file-write semantics

On `'stable-active` outcome of PROMOTE-STABLE:

1. Compute provenance bundle (cand_id, motif, workload_hash,
   coupling counts, energy delta, held-out wins, attestation row).
2. Write `cand_<NNN>.6th` containing:
   ```
   \ AUTO-GENERATED by PROMOTE-STABLE on <timestamp>
   \ DO NOT EDIT.  Source of truth is .lineage.json.
   \ Workload: <workload_hash>
   \ Held-out: <wins>/<total>
   \ Source tag: <fixture|demo|production>
   : cand_<NNN>  <motif body> ;
   ```
3. Write `cand_<NNN>.lineage.json` (Element 4 schema).
4. Initialize `cand_<NNN>.metabolism.json` with empty history.
5. Append `'promote-stable` event to `attestations/ledger.txt`
   (Element 3 schema).
6. Mark cand as persisted in current session's law_state.

If ANY of these steps fails, the entire promotion is rolled back
atomically (existing `ROLLBACK-STABLE` mechanism extended).
Partial writes are not permitted.

### Element 3 — Ledger promote-event schema

`attestations/ledger.txt` currently records pre-reg attestations:

```
<iso-ts>  <sha256-of-predictions-file>  <repo-path>  <git-head>
```

Cycle 35 extends the format to also record promote events:

```
<iso-ts>  promote-stable  <cand_id>  <lineage-sha256>  <git-head>  <source-tag>
```

The `lineage-sha256` is sha256 of the cand_NNN.lineage.json
content.  Reviewer can re-shasum the lineage file and verify the
ledger entry was not tampered with.

The two row formats are distinguished by column 2:
- column 2 is a hex hash → pre-reg attestation
- column 2 is the literal string `promote-stable` → promote event

### Element 4 — Lineage JSON schema (binding)

```json
{
  "cand_id": "cand_042",
  "discovery_cycle": 35,
  "discovered_at": "2026-NN-NNT...Z",
  "engine_version": "<git_sha at discovery time>",

  "source_tag": "production",                // fixture | demo | production
  "workload": {
    "source": "stress-test|application|exploration|...",
    "hash": "sha256:<workload_substrate_hash>",
    "ops_count": <int>,
    "seed": "<rng_seed_if_applicable>"
  },

  "motif": {
    "body": ["MARK", "drop", "NODES"],
    "length": 3,
    "auto_detected": true,                    // false for hand-authored = invalid
    "detection_window_position": <int>        // from DETECT-MOTIF-AUTO
  },

  "gates": {
    "shadow_check": {"pass": true, "world_delta_match": true, "cost_within_bound": true},
    "coupling": {"uses": 7, "sessions": 4, "passed": true},
    "energy_gate": {"delta_e": -12, "passed": true},
    "held_out_eval": {"wins": 5, "total": 6, "passed": true,
                      "substrate_set_hash": "sha256:..."},
    "promote_stable": {"status": "stable-active", "ledger_row_ts": "..."}
  },

  "attestation": {
    "ledger_row_index": <int>,
    "external_anchor": "tag-... or ots-... or null"
  }
}
```

Schema is binding.  Missing required field → cand rejected at
boot-time load with `'lineage-incomplete` error; cand_<NNN>.6th
is moved to `_rolled-back/`.

### Element 5 — Cross-boot loading rules

On engine boot (after install-meta-runtime!, before any user code):

1. Scan `stdlib/promoted/**/cand_*.6th`.
2. For each, load sibling `.lineage.json`.
3. Validate lineage: schema complete, source_tag ∈
   {`'fixture`, `'demo`, `'production`}, motif auto_detected=true,
   gates all passed.
4. If validation passes: register cand in law_state with its
   original cand_NNN identifier and motif body.
5. Load sibling `.metabolism.json` into recent counter alists
   (carry-over history, NOT reset).
6. If validation fails: move all three files to `_rolled-back/`
   with timestamp; record `'lineage-rejected` ledger event.

**Iron rule preservation:** boot-time loading MUST NOT trigger
`HELD-OUT-EVAL`.  Lineage records the original held-out result;
re-evaluating would violate the cycle 25 §11 iron rule.

### Element 6 — Metabolism state persistence

`cand_<NNN>.metabolism.json`:

```json
{
  "cand_id": "cand_042",
  "last_updated_at": "...",
  "last_updated_session_id": "<digest>",
  "epoch_counter_at_save": <int>,
  "momentum_history": [<m_at_t>, <m_at_t-1>, ...],   // up to MOMENTUM-HISTORY-WINDOW
  "current_status": "stable-active|stale|demotion-candidate|...",
  "heldout_wins_running": <int>,
  "positive_epochs_running": <int>,
  "observed_dependents_last_session": [<cand_id>, ...]
}
```

On `NEW-EPOCH`, after Pass C, the engine writes the updated
metabolism.json for every active promoted cand (atomic write
via temp-rename).

On engine boot, the loaded metabolism continues from saved
state — a cand that was 'demotion-candidate at last shutdown
remains 'demotion-candidate at boot; an idle gap counts as
contributing-zero epochs (NOT as positive evidence).

### Element 7 — Fixture / demo / production tagging

`INDUCE-RUNTIME` signature extended:

```
INDUCE-RUNTIME ( motif source-tag -- cand-id )
```

Where `source-tag` ∈ {`'fixture`, `'demo`, `'production`}.

Tag semantics:
- `'fixture`: cand dies at session end; NEVER written to
  `stdlib/promoted/`.  PROMOTE-STABLE on a fixture-tagged cand
  returns `'rejected-fixture-tag` without writing files.
- `'demo`: may be promoted but `source_tag = "demo"` in lineage;
  EXCLUDED from L2 substrate-discovered tally.
- `'production`: promoted cands count toward L2.  Requires
  workload to be `'application` or `'exploration` or `'stress-test`
  (NOT `'demo` or `'fixture`).

**Backwards compatibility:** existing demos that call
`INDUCE-RUNTIME` without a tag get `source-tag = 'fixture` by
default.  Demos 143-176 continue to pass without modification.

A retrofit pass MAY be performed to add explicit `'fixture` tags,
but is not required for cycle 35 PASS.

### Element 8 — Blind discovery harness

`stdlib/harness/blind-discovery.6th` (new file): a substrate
workload generator that the engineer does NOT design to trigger
a specific motif.

Minimum requirements:
- Configurable via env vars: `BLIND_HARNESS_OPS=N`,
  `BLIND_HARNESS_SEED=K`, `BLIND_HARNESS_GENERATOR=...`
- Uses `stdlib/rand.6th` LCG with explicit seed; reproducible.
- Generates substrate operations from a distribution that does
  NOT target any specific repetition pattern.
- Tags all `INDUCE-RUNTIME` calls during the run with
  `'production`.
- Runs through normal metabolism cycles (NEW-EPOCH every K ops).
- Reports: total ops executed, motifs auto-detected, INDUCE
  events, COMMIT-PRIMITIVE events, PROMOTE-STABLE events, files
  written.

Pass criterion (Element 9): at least one blind harness run is
executed and its output is recorded in `RESULTS-180-cycle35.md`
(or similar), even if it yields zero production discoveries.

### Element 9 — PASS / STRONG PASS / VALID NEGATIVE / FAIL criteria

**PASS** (all required):

1. `stdlib/promoted/<cycle>/cand_<NNN>.6th` files are created
   ONLY by PROMOTE-STABLE.  Manual file creation cannot make a
   cand appear in L2.
2. Every promoted cand has `.lineage.json` matching the binding
   schema (Element 4).
3. `attestations/ledger.txt` contains the `'promote-stable`
   event for each promoted cand (Element 3 format).
4. Engine boot reloads promoted candidates per Element 5 rules.
5. Metabolism state survives at least one reboot (a cand still
   has its prior momentum history available in the next session).
6. `'fixture` / `'demo` / `'production` tags are enforced per
   Element 7.
7. Fixture and demo candidates CANNOT be counted as L2
   regardless of how many PROMOTE-STABLE calls succeed.
8. At least one blind harness run (Element 8) is executed and
   results recorded, even if it yields zero production
   discoveries.

**STRONG PASS** (all of PASS, plus):

- At least one `'production`-tagged cand_NNN survives at least
  one cross-boot metabolism cycle.

**VALID NEGATIVE RESULT** (all of PASS minus the production-cand
existence, plus):

- Blind harness produces zero durable discoveries, but the
  persistence machinery works mechanically.  Verifiable by
  intentionally constructing a `'production` cand via test
  fixture and confirming it would persist if the harness had
  found one.

A valid negative result is a legitimate PASS — it means the
machinery is built, the protocol works, the workload just didn't
happen to produce a survivor.  This is honest research.

**FAIL** (any one triggers):

1. Demo candidates are written into `stdlib/promoted/` without
   `source_tag = "demo"` in lineage (i.e., tagged as
   `'production` falsely).
2. cand_NNN files can be hand-authored (no PROMOTE-STABLE
   provenance) and counted as discovered.
3. Lineage is missing for any cand in `stdlib/promoted/`.
4. Ledger has no `'promote-stable` event for any cand in
   `stdlib/promoted/`.
5. Boot reload is fake (e.g., just touches the file but doesn't
   register in law_state) or works only in test harness.
6. Metabolism resets on reboot (cands re-enter with empty
   momentum history).
7. Cycle 34A (external energy) is implemented before cycle 35
   PASS or STRONG PASS.

### Element 10 — Negative tests preventing demo leakage into L2

These tests must be added as cycle 35 demos and must PASS as
part of regression:

- **NEG-1 — Fixture-tagged INDUCE never writes to stdlib/promoted/.**
  Run a demo that INDUCEs with `'fixture` tag, drives the cand
  through coupling + energy + held-out, calls PROMOTE-STABLE.
  Assert: PROMOTE-STABLE returns `'rejected-fixture-tag`;
  `stdlib/promoted/` contains no new file.

- **NEG-2 — Hand-written cand_NNN file without lineage is
  rejected at boot.**  Manually create
  `stdlib/promoted/_test/cand_999.6th` (no .lineage.json).
  Boot engine.  Assert: cand_999 is NOT in active dictionary;
  file moved to `_rolled-back/`; `'lineage-rejected` event in
  ledger.

- **NEG-3 — Lineage with workload_source='demo' excluded from
  L2 tally.**  Construct lineage with `source_tag = "demo"`.
  Confirm L2 count (via inspection primitive) does not include it.

- **NEG-4 — Boot reload does NOT trigger HELD-OUT-EVAL.**  Boot
  engine with a promoted cand whose lineage records held-out
  result.  Assert: no new held-out evaluation runs; iron rule
  preserved.

- **NEG-5 — Manually-edited cand_NNN.6th body is detected.**
  After PROMOTE-STABLE, manually modify the cand body in the
  .6th file.  Boot engine.  Assert: detect body-vs-lineage
  mismatch; reject cand; move to `_rolled-back/`; ledger event.

- **NEG-6 — Cycle 34A primitives unavailable until L2 non-empty.**
  Attempt to call `INJECT-ENERGY` (cycle 34A prim).  Assert: word
  is unbound or returns `'cycle-35-pre-requisite-unmet` until
  cycle 35 has at least one production cand persisted across
  reboots.  (This is the operational enforcement of CLAIM-5.)

## Main pitfall (binding warning)

> Making the `stdlib/promoted/` directory and writing some files
> there does NOT make L2 appear.

L2 occupancy requires the **full lineage chain**:

```
blind workload (engineer-blind!)
  → DETECT-MOTIF-AUTO finds a motif
  → SHADOW-CHECK passes
  → coupling gate (N=5, M=3) passes
  → energy gate (delta_e < 0) passes
  → HELD-OUT-EVAL ≥ 4/6 wins
  → PROMOTE-STABLE writes cand_<NNN>.6th + lineage + ledger event
  → engine reboot
  → boot reload validates lineage + loads cand + restores metabolism
  → next session's metabolism observes the cand
  → cand survives (or honestly decays through normal gates)
```

A cand that skips ANY step is not a discovery.  A cand whose
workload was hand-crafted is not a discovery.  A cand that exists
only inside one test process is not a discovery.

This warning must appear in `RESULTS-180-cycle35.md` (the eventual
results file) as a sanity check against future-self optimism.

## What this pre-reg does NOT claim

- That the persistence layer will produce a non-empty L2 on
  first try.  VALID NEGATIVE is an acceptable PASS.
- That the discovery-quality question is answered.  Cycle 35
  enables discovery to exist; quality evaluation is later cycles.
- That the existing protocol primitives (cycles 25-34) need
  to change.  Cycle 35 adds persistence around them, does not
  modify them.
- That cycle 35 is the only path to L2 occupancy.  An alternative
  (e.g., loading manually-verified promoted cands from a curated
  manifest) might also populate L2, but would be a different
  semantics; cycle 35 is the substrate-discovered path.
- That this is the final design.  Schemas (lineage.json,
  metabolism.json) may need v2 if later cycles discover gaps.

It claims only:

1. **Three invariants binding for L2:** lineage / production tag /
   cross-boot survival.
2. **Ten elements binding for implementation:** layout, write
   semantics, ledger schema, lineage schema, load rules,
   metabolism persistence, tagging, blind harness, pass/fail,
   negative tests.
3. **L2 appears only when the full chain works,** not when the
   directory exists.

If cycle 35 lands with these in place, L2 becomes a category
that CAN have occupants.  Whether it WILL is a separate question
the blind harness will answer empirically.
